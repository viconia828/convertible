# 2026-04-22 strategy 单交易日观察导出方案

## 背景

截至 `2026-04-22`，`strategy` 已经具备：

- 单日期 `snapshot -> env/factors -> weights -> portfolio` 主链
- 终端预览入口
- 观察名单视角
- `summary / verbose` 双视图
- runtime snapshot reuse
- 统一 cache diagnostics 展示

但当前入口仍主要是终端预览。用户如果想像“环境打分 / 因子打分”那样，把某个交易日的运行结果导出成 Excel 观察文件，目前还没有对应入口。

## 目标

本轮新增一个“单交易日 strategy 观察导出”入口，满足：

1. 用户输入一个交易日。
2. 可选输入一组观察名单代码。
3. 输出一个 XLSX，便于离线查看和留档。
4. 导出内容应覆盖单日策略结果的核心观察面。

## 范围

本轮纳入：

- 新增 `strategy_exports.py`
- 新增 `build_strategy_observation_report(...)`
- 新增 `write_strategy_observation_xlsx(...)`
- 新增 `tools/export_strategy_observation.py`
- 新增批处理入口
- 新增测试

本轮不做：

- 不改现有 `preview_strategy.py` 的终端职责
- 不扩展到批量日期导出
- 不扩展到正式回测输出
- 不新增磁盘级 strategy snapshot cache

## 设计原则

### 1. 终端预览和 Excel 导出分开

已有 [preview_strategy.py](C:/Users/ai/Desktop/可转债多因子/tools/preview_strategy.py) 是“终端快速看”入口。

本轮新增的观察导出是“文件观察 / 留档”入口。

两者都复用同一条 `StrategyService -> StrategyEngine` 主链，但不要让：

- 终端预览开始承担 Excel 导出职责
- Excel 导出反过来依赖 tool 层私有渲染逻辑

### 2. 导出层新建独立模块

参考现有环境 / 因子导出模式，本轮新增 [strategy_exports.py](C:/Users/ai/Desktop/可转债多因子/exports/strategy_exports.py)，负责：

- report dataclass
- report build
- xlsx write

而不是把这部分塞回 [scoring_exports.py](C:/Users/ai/Desktop/可转债多因子/exports/scoring_exports.py)。

原因：

- `strategy` 已经是独立模块
- 环境/因子导出属于更底层导出能力
- `strategy` 导出应该作为独立导出层，而不是继续混在“打分导出”文件里

## 导出内容

建议工作簿包含以下 sheet：

1. `environment`
   - 单日环境分数
2. `factor_weights`
   - 单日因子权重
3. `portfolio`
   - 目标组合
4. `watchlist`
   - 如果指定观察名单，则输出观察名单视角表；未指定时输出空表头
5. `candidate_scores`
   - 单日全市场候选总分、eligible、exclude_reason 等核心字段
6. `factor_diagnostics`
   - 单日因子原始值 / 分数 / 诊断列
7. `cache_diagnostics`
   - 单次运行 cache diagnostics 摘要
8. `run_summary`
   - 运行摘要、参数、数据质量状态、notes

## 为什么要包含 candidate_scores / factor_diagnostics

如果只导出环境、权重和组合，用户只能看到“最后入选了谁”，但看不到：

- 为什么某个债没有入选
- 它是根本不 eligible，还是可建仓但没挤进 top_n
- 五因子分和 raw 因子值大概落在哪

因此单日观察导出需要同时保留：

- 面向最终结果的 `portfolio`
- 面向比较观察的 `candidate_scores`
- 面向排查的 `factor_diagnostics`

## watchlist 视角

如果用户输入观察名单，则导出一张更聚焦的表，建议保留：

- `代码`
- `观察状态`
- `可建仓`
- `候选排序`
- `组合排序`
- `目标权重`
- `总分`
- `剔除原因`

如果没输入观察名单，也仍然保留空 sheet，便于工作簿结构稳定。

## cache diagnostics 导出

既然终端预览已经有缓存诊断，本轮导出也应一起保留。

但这里不导出原始 `stats_snapshot()` 大 dict，而导出已经收口后的结构化摘要：

- 统一 `hit / miss / partial hit / remote fill / file scan / writeback`
- `request panel / aggregate / aggregate metadata`
- top stages
- `runtime_snapshot_reused`

这样用户在离线查看单日观察文件时，也能直接判断这次运行的大致缓存行为。

## 输出文件命名

沿用现有导出风格：

- 放在配置里的 `exports.output_dir`
- 文件名前缀固定为 `策略观察`
- 日期 token 使用请求交易日

即：

- `策略观察_20260422_20260422_时间戳.xlsx`

## CLI 设计

新增 [tools/export_strategy_observation.py](C:/Users/ai/Desktop/可转债多因子/tools/export_strategy_observation.py)：

## 测试

至少覆盖：

1. report build 能正确透传单日 decision 的核心结果
2. watchlist sheet 能生成预期列结构
3. xlsx 能写出约定的 sheet 集合
4. tool 入口能走交互式 / CLI 路径

## 风险与取舍

### 1. 不复用 preview 的文本渲染

Excel 导出和终端渲染面对的是两种不同介质。

因此本轮只复用底层决策结果，不复用 `render_strategy_preview(...)`。

### 2. 不在本轮做批量日期

单日观察和批量回放的关注点不一样。

本轮先把“单日可观察文件”做稳，再考虑是否有必要补多日期 strategy 导出。
