# 2026-04-24 缓存 schema / 口径版本治理实现记录

## 本轮目标

- 不给所有缓存 CSV 一次性引入新的 sidecar 版本文件。
- 先治理当前已经存在的两类 metadata：
  - `time_series/<dataset>/<cache_key>.coverage.json`
  - `time_series_aggregate/<dataset>/<profile>/<partition>.meta.json`
- 让缓存层先具备三个能力：
  - 识别标准化 schema 变化
  - 识别 aggregate 投影列变化
  - 对缺少治理块的老 metadata 自动失效，而不是误命中

### 1. 为标准化数据集增加稳定 schema 签名

- 新增 [data/cache/governance.py](C:/Users/ai/Desktop/可转债多因子/data/cache/governance.py)。

- 当前治理块统一包含：
  - `metadata_format_version`
  - `dataset_name`
  - `standardized_name`
  - `schema_signature`
- 其中 aggregate metadata 还额外包含：

### 3. `coverage.json` 读写已经纳入治理

- [data/cache/coverage.py](C:/Users/ai/Desktop/可转债多因子/data/cache/coverage.py) 中的 `load_time_series_coverage` / `save_time_series_coverage` 已改成通过治理 helper 读写。
- 现在如果旧 `coverage.json` 缺少 `governance`，会直接返回 `None`，让上层走补数或重建逻辑。

### 4. aggregate `.meta.json` 读写已经纳入治理

- `load_time_series_aggregate_metadata` / `save_time_series_aggregate_metadata` 已改成通过治理 helper 读写。
- 当前 aggregate metadata 会同时校验：
  - governance 基本字段是否匹配
  - `schema_signature` 是否仍和当前标准化 schema 一致
  - `projection_signature` 是否与 metadata 里的 `projection_columns` 一致
  - 当前请求列是否仍是已缓存投影列的子集

### 5. `DataLoader` 已接入投影兼容校验

- [data/data_loader.py](C:/Users/ai/Desktop/可转债多因子/data/data_loader.py) 在读取 `cb_daily_cross_section` 月聚合缓存时，已经把 `requested_columns` 传给 metadata 读取链路。
- 现在如果：
  - 聚合 metadata 缺少治理块
  - schema 已变化
  - 投影签名不一致
  - 本次请求列超出旧聚合投影

都会直接把该月 aggregate 视为失效，回退到日缓存 / 重建路径。

## 当前行为结论

- 老的无治理 metadata 不会再被静默复用。
- aggregate metadata 被手工改坏时，不会再因为文件存在就被误当成有效。
- 旧聚合只缓存了窄列投影时，新的宽列请求会自动触发重建，不会把旧 aggregate 错当成命中。
- 这一版还没有给所有原始缓存 CSV 增加统一 sidecar；当前治理范围只覆盖已有 metadata。

- `cb_daily_cross_section` 在投影变宽时应绕过旧 aggregate，改走重建路径

## 下一步建议

- 先不要急着给所有缓存文件上统一 sidecar，优先把“真正会影响回测稳定性”的预热与失效策略继续收口。
- 可以考虑把 `reference metadata` 和后续新增的聚合 metadata 也接到同一治理口径里，但这一步的优先级低于回测主链准备。
- 进入批量回测前，建议先补一轮：
  - 预热窗口约束
  - cache invalidation 边界
  - runtime reuse 与落盘缓存之间的职责边界
