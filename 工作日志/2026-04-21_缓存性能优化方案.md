# 2026-04-21 缓存性能优化方案

## 1. 背景

统一缓存层第一阶段已经完成，但真实样本 benchmark 表明，缓存层收口之后，因子打分链路仍然存在明显的缓存读取浪费。

在当前热缓存场景下，运行：

- `python tools\benchmark_factor_export.py --start-date 2025-10-01 --end-date 2026-04-17 --top-codes 2 --label cache_perf_baseline --skip-write`

得到基线结果：

- 两只转债、半年窗口、纯构建耗时约 `10.7s`

进一步查看缓存读写计数后发现：

- `grouped_time_series_file_reads = 664`

说明真正的瓶颈已经不再是远端取数，而是本地缓存读取和批量拼接。

## 2. 已确认的热点

当前 `build_factor_score_report(...)` 中，`cb_rate` 的读取策略存在明显过量加载：

1. 只要请求窗口内存在任意 `ytm` 缺失，就会触发 `cb_rate` 读取
2. 一旦触发，当前实现会把请求窗口内全市场所有出现过的可转债代码都传给 `get_cb_rate(...)`
3. 但 `FactorEngine` 实际只会对“当前快照里 `ytm` 缺失的债”调用 `estimate_ytm_series(...)`

在真实样本里已经确认：

- 请求窗口内全市场代码数：`450`
- 实际存在 `ytm` 缺失的代码数：`2`

也就是说，目前为了给极少数缺失 `ytm` 的债补值，额外读取了数百份 `cb_rate` 缓存文件。

## 3. 本轮优化目标

本轮先做一刀高收益、低风险优化：

- 将 `cb_rate` 从“窗口内全市场代码加载”改为“仅对请求窗口内存在 `ytm` 缺失的代码加载”

同时补充最小必要的观测能力，便于继续追踪缓存热点：

- 在缓存层保留更细粒度的 dataset 级 grouped file read 统计
- 在 benchmark / 验证脚本中能直接看到优化前后差异

## 4. 设计方案

### 4.1 `cb_rate` 按缺失 `ytm` 代码按需加载

在 [scoring_exports.py](C:/Users/ai/Desktop/可转债多因子/exports/scoring_exports.py) 中：

- 先筛出请求窗口内 `ytm` 为缺失值的行
- 只提取这些行对应的唯一 `cb_code`
- 仅当缺失代码列表非空时才调用 `loader.get_cb_rate(...)`

这样可以保证：

- 不改变因子计算口径
- 不影响 `ytm` 缺失时的补值逻辑
- 只减少不必要的缓存读取

### 4.2 缓存层补充更细粒度统计

在 [data/cache/service.py](C:/Users/ai/Desktop/可转债多因子/data/cache/service.py) 中：

- 为 grouped cache load 增加 dataset 级统计项
- 例如：
  - `grouped_time_series_file_reads`
  - `grouped_time_series_file_reads::cb_daily_cross_section`
  - `grouped_time_series_file_reads::cb_rate`

这样后续 benchmark 不需要重新 profile，就能快速看出具体是哪个 dataset 在吞缓存读取时间。

## 5. 实施范围

本轮只做以下范围内的改动：

- `build_factor_score_report(...)` 的 `cb_rate` 读取策略
- `DataCacheService` 的 grouped cache read 统计
- 相关单元测试
- benchmark / 验证输出

本轮不做：

- 聚合缓存格式升级
- `parquet` 改造
- 独立 derived cache
- 运行期内存缓存

这些继续保留在统一缓存层设计文档中的后续优化分支里。

## 6. 验证标准

完成后应满足：

1. 现有因子打分结果不变
2. `cb_rate` 只对缺失 `ytm` 的代码发生读取
3. 回归测试继续通过
4. 真实样本 benchmark 的缓存读取次数明显下降
5. 纯构建耗时相对基线进一步下降

## 7. 本轮实施结果

本轮已经按方案完成实现并验证：

- `scoring_exports.py` 已改为仅对请求窗口内 `ytm` 缺失代码加载 `cb_rate`
- `DataCacheService` 已增加 dataset 级缓存读写统计
- `benchmark_factor_export.py` 已能直接输出 `cache_stats`

验证结果：

- 回归：
  - `python -m unittest tests.test_cache_service tests.test_scoring_exports tests.test_data_loader tests.test_factor tests.test_derived_metrics -v`
  - `38` 个测试全部通过

- 真实样本 benchmark：
  - 优化前：两只转债、`2025-10-01 ~ 2026-04-17`、纯构建约 `10.7s`
  - 优化后：同窗口、纯构建约 `6.8s`
  - `grouped_time_series_file_reads` 从 `664` 降到 `214`

结论：

- 这轮优化已经确认有效
- 当前剩余缓存热点将更多集中在 `cb_daily_cross_section` 的按日文件读取本身，而不是 `cb_rate` 的过量加载
