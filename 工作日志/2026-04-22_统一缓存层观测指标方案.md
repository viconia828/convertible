# 2026-04-22 统一缓存层观测指标方案

## 1. 背景

当前统一缓存层已经有一批分散统计：

- `aggregate_memory_hit_calls / miss_calls`
- `aggregate_metadata_memory_hit_calls / miss_calls`
- `panel_memory_hit_calls / miss_calls / save_calls / invalidation_calls`
- `grouped_time_series_file_reads`
- `derived_writeback_files`

这些统计已经足够支撑前两轮缓存优化落地，但还不够回答下面这些更统一的问题：

- 这次请求整体算 cache hit、miss，还是 partial hit
- 本轮到底扫描了多少缓存文件
- 这次实际触发了多少次远端补数
- 热点链路到底写回了多少次缓存
- 时间主要花在本地查缓存、远端补数，还是结果装配

而 [2026-04-21_统一缓存层设计方案.md](C:/Users/ai/Desktop/可转债多因子/工作日志/2026-04-21_统一缓存层设计方案.md) 和最近两天的下一步清单，已经把“统一缓存层观测指标”明确列为下一步主线。

## 2. 目标

本轮目标是补一层**统一口径的缓存观测指标**，并先接到当前最热的几条 `DataLoader` 路径上。

完成后至少要能统一看到：

1. `hit / miss / partial hit`
2. 文件扫描数
3. 远端补数次数
4. 回写次数
5. 分阶段耗时

同时保留现有更细的旧统计，不做破坏性替换。

## 3. 非目标

本轮不做：

- 完整的 UI / 终端缓存诊断面板
- 覆盖所有冷门 dataset 的全量打点
- 单次请求级 trace id
- 独立的 observability 模块拆分
- 缓存命中收益的最终可视化报表

## 4. 设计判断

### 4.1 统一语义放在 `DataCacheService`

虽然 `hit / miss / partial hit` 的判断发生在 `DataLoader`，但统计本身仍应收口在 [service.py](C:/Users/ai/Desktop/可转债多因子/data/cache/service.py)。

因此本轮做法是：

- 在 `DataCacheService` 增加统一观测记录接口
- 让 `DataLoader` 在关键决策点调用这些接口

这样业务层仍不直接操作观测字典。

### 4.2 保留旧统计，新增统一统计

例如：

- request panel 的 `panel_memory_hit_calls` 继续保留
- 同时新增更上层的 `cache_resolution_hit_calls`

两层统计并存：

- 旧统计回答“某个具体缓存层有没有命中”
- 新统计回答“这次请求整体属于哪种缓存结果”

### 4.3 第一批接入范围

本轮优先接入以下热点路径：

- `get_cb_basic(...)`
- `_get_time_series(...)`
  - 间接覆盖 `cb_daily / stock_daily / index_daily`
- `get_cb_daily_cross_section(...)`
- `get_cb_rate(...)`
- `get_cb_call(...)`
- `get_yield_curve(...)`
- `get_cb_equal_weight_index(...)`

原因是这些路径已经覆盖了当前环境、因子、strategy 预览主链的大部分热点。

## 5. 指标设计

### 5.1 请求结果口径

新增统一统计：

- `cache_resolution_hit_calls`
- `cache_resolution_miss_calls`
- `cache_resolution_partial_hit_calls`
- `cache_resolution_refresh_bypass_calls`

语义：

- `hit`
  - 本次请求所需数据全部由本地缓存解决
- `miss`
  - 本次请求完全没有得到有效缓存覆盖，需要远端补数
- `partial_hit`
  - 本地缓存覆盖了一部分，但仍需远端补数
- `refresh_bypass`
  - 本次因显式 `refresh=True` 绕过缓存结果判断

### 5.2 文件扫描

新增统一统计：

- `cache_file_scan_calls`

语义：

- 每次尝试读取缓存文件或 metadata 文件时计数
- 批量 grouped 读取按逐文件计数

### 5.3 远端补数

新增统一统计：

- `remote_fill_calls`

语义：

- 每次实际发起一次远端查询时计数
- 第一版按“调用次数”记录，不按返回行数记录

### 5.4 回写

新增统一统计：

- `cache_writeback_calls`

语义：

- 每次写盘一个缓存文件或 metadata 文件时计数
- runtime-only memory cache 不计入该项

### 5.5 分阶段耗时

新增统一统计：

- `stage_calls`
- `stage_elapsed_ms`

阶段名第一版建议包括：

- `cache_lookup`
- `request_panel_lookup`
- `aggregate_lookup`
- `day_cache_lookup`
- `remote_fetch`
- `aggregate_materialize`
- `result_concat`
- `result_filter`
- `rebuild_from_cross_section`

耗时按毫秒累计到 stats 中，保持与现有 `stats_snapshot()` 的整数口径一致。

## 6. 对外输出

除了继续保留 `stats_snapshot()` 原始扁平统计外，本轮新增：

- `DataCacheService.observability_snapshot()`

输出更易读的聚合摘要：

- hit / miss / partial hit / refresh bypass 总计
- file scan / remote fill / writeback 总计
- `stage_elapsed_ms`
- `stage_calls`

并在 benchmark 工具中直接附带这份摘要，方便真实样本验证。

## 7. 测试计划

### 7.1 `DataCacheService`

补直接测试：

- `observability_snapshot()` 能汇总统一统计
- grouped 文件读取会同步累计 `cache_file_scan_calls`
- 写回流程会累计 `cache_writeback_calls`

### 7.2 `DataLoader`

补热点路径测试：

- 局部命中的 code/day 请求会记录 `partial_hit`
- 完全命中的重复请求会记录 `hit`
- 需要远端补数时会记录 `remote_fill_calls`
- 主热点路径会留下 `stage_elapsed_ms::*` 统计

## 8. 验收标准

1. 统一缓存层能输出 hit / miss / partial hit / refresh bypass 总计。
2. 热点路径能输出文件扫描数、远端补数次数、回写次数。
3. 至少 `cb_daily`、`cb_rate`、`cb_daily_cross_section` 三条主链能输出阶段耗时。
4. benchmark 输出中能直接看到统一观测摘要。
5. 现有缓存与 strategy 子集回归继续通过。
