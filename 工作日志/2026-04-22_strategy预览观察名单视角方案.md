# 2026-04-22 strategy 预览观察名单视角方案

## 1. 背景

截至 `2026-04-22`，`strategy` 单日期预览入口已经可用：

- [tools/preview_strategy.py](C:/Users/ai/Desktop/可转债多因子/tools/preview_strategy.py)
- [strategy/service.py](C:/Users/ai/Desktop/可转债多因子/strategy/service.py)
- [strategy/engine.py](C:/Users/ai/Desktop/可转债多因子/strategy/engine.py)

当前入口已经能输出：

- 当日环境分数
- 当日因子权重
- 目标组合
- 对齐摘要与提示信息

但它仍然偏“看全量结果”。对于日常联调和实盘观察，更常见的需求其实是：

- 我关心几只候选债今天有没有入选
- 没入选的话，是因为不可交易、分数不够，还是当日根本没有横截面记录
- 我不想先读完整个全市场组合，再手工定位这些代码

因此，本轮为 `strategy` 预览入口补一个“指定候选代码 / 观察名单”视角。

## 2. 目标

本轮目标不是改变 `strategy` 的计算口径，而是给预览入口补一个更适合日常联调的聚焦视图。

用户体验目标：

1. 用户仍然指定一个交易日运行单日期预览。
2. 用户可以可选地再输入一组转债代码。
3. 预览结果仍然保留全市场口径算出来的环境、权重和目标组合。
4. 终端额外补一张“观察名单视角”表，直接展示这些代码的当日状态。

## 3. 设计边界

### 3.1 不改变全市场横截面口径

这轮最重要的边界是：

- `requested_codes` 只影响预览展示
- 不影响环境计算
- 不影响因子标准化
- 不影响组合构建

也就是说，策略仍然基于全市场横截面计算分数和构建组合，而不是把观察名单当成一个小样本池自比。

这与既有设计约束保持一致：

- [2026-04-21_strategy模块骨架设计方案.md](C:/Users/ai/Desktop/可转债多因子/工作日志/2026-04-21_strategy模块骨架设计方案.md)

其中已经明确：`requested_codes` 只能控制最终展示或裁切，不能改变全市场标准化口径。

### 3.2 不提前做批量预览或回测特性

本轮只做单日期预览入口的增强，不扩展到：

- 多日期批量预览
- 回测主链路
- Excel 导出
- 正式调仓执行

### 3.3 不新增复杂观察名单文件格式

本轮先只支持：

- CLI `--codes`
- 交互式手工输入

暂不做：

- `--codes-file`
- 持仓文件读取
- 外部观察名单配置文件

## 4. 入口设计

建议在 [tools/preview_strategy.py](C:/Users/ai/Desktop/可转债多因子/tools/preview_strategy.py) 中新增可选参数：

- `--codes`

语义：

- 输入一个或多个可转债代码
- 支持空格、逗号、分号分隔
- 输入六位代码时自动补全交易所后缀

交互式模式下，新增一个可选输入项：

- `可选，请输入候选代码 / 观察名单`

如果留空，则行为与当前版本一致。

## 5. 数据契约设计

当前 `StrategyService.build_snapshot(...)` 已经有 `requested_codes` 参数，`StrategySnapshot` 里也已有对应字段。

本轮建议做两件事：

1. 把 `requested_codes` 的规范化做完整
2. 把规范化后的结果透传到 `StrategyDecision.diagnostics`

规范化规则建议对齐现有导出入口习惯：

- 去重
- 去空白
- 全部转大写
- 六位数时自动补全交易所后缀
  - `11xxxx -> .SH`
  - `12xxxx -> .SZ`

这样一来，CLI 渲染层无需自己维护第二套代码规范化逻辑。

## 6. 观察名单视角输出设计

建议在终端输出中新增一个独立区块：

- `观察名单视角`

每个请求代码至少展示：

- `代码`
- `观察状态`
- `可建仓`
- `候选排序`
- `组合排序`
- `目标权重`
- `总分`
- `剔除原因`

### 6.1 状态定义

建议统一收敛成下面四类：

1. `已入选目标组合`
2. `可建仓未入选`
3. `不可交易/已剔除`
4. `当日无横截面记录`

这样用户一眼就能分清：

- 是不是有记录
- 能不能进候选池
- 有没有真的进组合

### 6.2 展示顺序

观察名单表建议按用户输入顺序展示，而不是按总分重排。

原因是这张表的目标是“定位指定代码”，不是再次展示市场排名。

## 7. 测试要求

本轮至少补下面几类测试：

1. `StrategyService`:
   - 原始字符串能被正确拆分和规范化
   - 六位代码能自动补全交易所后缀
2. `StrategyEngine`:
   - `requested_codes` 能透传到 diagnostics
3. `preview_strategy`:
   - `--codes` 能传入 `StrategyService`
   - 观察名单视角能正确显示四类状态
   - 交互式模式能读取可选观察名单输入

这样既能直接改善日常使用体验，又不会把范围过早拉进回测或执行层。
