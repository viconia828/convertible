# 2026-04-21 strategy 模块骨架设计方案

## 1. 背景

截至 `2026-04-21`，项目的“数据层 + 环境打分 + 因子打分 + 缓存/性能优化”已经基本收口，现阶段最适合进入下一条主线：

`snapshot -> env/factors -> weights -> portfolio`

当前已经可直接复用的能力有：

- [data/data_loader.py](C:/Users/ai/Desktop/可转债多因子/data/data_loader.py)
- [data/cache/service.py](C:/Users/ai/Desktop/可转债多因子/data/cache/service.py)
- [env/environment_detector.py](C:/Users/ai/Desktop/可转债多因子/env/environment_detector.py)
- [factor/factor_engine.py](C:/Users/ai/Desktop/可转债多因子/factor/factor_engine.py)
- [model/weight_mapper.py](C:/Users/ai/Desktop/可转债多因子/model/weight_mapper.py)
- [strategy_config.py](C:/Users/ai/Desktop/可转债多因子/config/strategy_config.py)

因此，`strategy` 第一阶段的目标不是重新发明一套环境/因子体系，而是把现有能力收成一个“可复用、可测试、可接回测”的策略骨架。

## 2. 本轮设计目标

本轮只设计 `strategy` 第一阶段骨架，目标是：

1. 定义 `strategy` 模块的职责边界。
2. 明确 `snapshot`、策略输入、策略输出的数据契约。
3. 设计“纯策略核心”和“数据协调层”的分层。
4. 给出第一阶段最小可落地文件结构。
5. 明确后续和 `risk / execution / backtest` 的衔接点。

## 3. 本轮不做什么

这轮设计先不把范围拉太大，下面这些能力不纳入第一阶段实现：

- 不实现回测引擎。
- 不实现执行层和真实持仓状态机。
- 不实现复杂风控和调仓约束。
- 不实现在线学习或自适应调权。
- 不实现新的缓存体系，继续复用当前统一缓存层。
- 不让 `strategy` 直接依赖导出层私有 helper。

## 4. 核心设计原则

### 4.1 策略核心保持纯函数

`StrategyEngine` 只做策略决策，不直接触碰：

- 文件系统
- 缓存读写
- Tushare / Chinabond 远端补数
- Excel 导出

也就是说，策略核心看到的应该是一个已经准备好的 `snapshot`，然后输出一个确定性的 `decision`。

### 4.2 数据协调和策略决策分层

建议拆成两层：

- `StrategyService`
  - 负责拉数据、走缓存、构造 snapshot、调用策略核心。
- `StrategyEngine`
  - 负责 `env -> factor weights -> total scores -> target portfolio`。

这样后面无论是 CLI 预览、回测、还是模拟盘，都可以复用同一个 `StrategyEngine`。

### 4.3 继续复用现有组件，不穿透到导出层

`strategy` 不应去调用 [scoring_exports.py](C:/Users/ai/Desktop/可转债多因子/exports/scoring_exports.py) 里的导出拼装逻辑。

建议复用的是底层计算能力：

- 环境计算：`EnvironmentDetector`
- 因子计算：`FactorEngine`
- 权重映射：`WeightMapper`
- 数据读取与缓存：`DataLoader + DataCacheService`

### 4.4 第一阶段先做“纯信号版 strategy”

第一阶段先把信号主链打通：

`snapshot -> env -> factor weights -> total scores -> portfolio`

先不在 `strategy` 内加入复杂风险控制、交易成本、持仓冷却期等逻辑。这些属于后续 `risk / execution / backtest` 阶段。

## 5. 建议模块结构

建议新增目录：

```text
strategy/
  __init__.py
  snapshot.py
  result.py
  portfolio.py
  engine.py
  service.py
```

各文件职责如下：

### 5.1 [strategy/snapshot.py](C:/Users/ai/Desktop/可转债多因子/strategy/snapshot.py)

定义策略所需的输入对象，建议使用 dataclass。

建议至少包含：

- `StrategySnapshot`
- `StrategyHistoryWindow`

`StrategySnapshot` 第一阶段建议字段：

- `trade_date`
- `history_start`
- `trade_calendar`
- `cb_daily`
- `macro_daily`
- `requested_codes`
- `data_quality_hints`

这里的 `cb_daily` 指向“截至 `trade_date` 的可转债历史横截面面板”，仍然走当前 `cb_daily_cross_section + factor_history_v1` 缓存链路。

### 5.2 [strategy/result.py](C:/Users/ai/Desktop/可转债多因子/strategy/result.py)

定义策略输出对象，建议至少包含：

- `StrategyDecision`
- `StrategyDiagnostics`

`StrategyDecision` 第一阶段建议字段：

- `trade_date`
- `environment`
- `factor_weights`
- `factor_scores`
- `total_scores`
- `eligible_count`
- `selected_portfolio`
- `diagnostics`

### 5.3 [strategy/portfolio.py](C:/Users/ai/Desktop/可转债多因子/strategy/portfolio.py)

负责把“单债总分”转换成目标组合。

第一阶段建议只提供简单、稳定、可解释的组合构建器，例如：

- `build_top_n_portfolio(...)`
- `normalize_capped_weights(...)`

默认组合逻辑建议：

1. 只在 `eligible == True` 的池子里选债。
2. 按 `total_score` 从高到低排序。
3. 取 `top_n`。
4. 权重生成先用简单的“分数比例 + 单债上限 + 归一化”。
5. 如果分数异常、候选不足或全相等，则回退到等权。

### 5.4 [strategy/engine.py](C:/Users/ai/Desktop/可转债多因子/strategy/engine.py)

策略核心，建议命名为 `StrategyEngine`。

建议接口：

```python
class StrategyEngine:
    def __init__(self, ...):
        ...

    def run(self, snapshot: StrategySnapshot) -> StrategyDecision:
        ...
```

内部步骤建议固定为：

1. 由 `snapshot` 计算环境分数。
2. 由 `snapshot` 计算因子分数。
3. 用 `WeightMapper` 计算当期因子权重。
4. 计算单债总分。
5. 构造目标组合。
6. 输出结构化 diagnostics。

### 5.5 [strategy/service.py](C:/Users/ai/Desktop/可转债多因子/strategy/service.py)

策略协调层，建议命名为 `StrategyService`。

建议接口：

```python
class StrategyService:
    def __init__(self, loader: DataLoader | None = None, ...):
        ...

    def build_snapshot(
        self,
        trade_date: str | pd.Timestamp,
        requested_codes: Sequence[str] | None = None,
        refresh: bool = False,
    ) -> StrategySnapshot:
        ...

    def run_for_date(
        self,
        trade_date: str | pd.Timestamp,
        requested_codes: Sequence[str] | None = None,
        refresh: bool = False,
    ) -> StrategyDecision:
        ...
```

`StrategyService` 可以读缓存、补数、构造 snapshot，但不负责导出 Excel。

## 6. 关键分层关系

建议采用下面这个分层：

```text
DataLoader / DataCacheService
  -> StrategyService
    -> StrategyEngine
      -> EnvironmentDetector
      -> FactorEngine
      -> WeightMapper
      -> PortfolioBuilder
```

职责边界：

- `DataLoader / DataCacheService`
  - 负责数据与缓存。
- `StrategyService`
  - 负责“把数据准备成策略可消费的 snapshot”。
- `StrategyEngine`
  - 负责纯策略决策。
- `PortfolioBuilder`
  - 负责把分数变成组合。

## 7. snapshot 的数据契约

第一阶段不建议把太多业务状态塞进 snapshot。先保持“决策所需最小集合”。

建议 `StrategySnapshot` 只包含：

- `trade_date`
- `history_start`
- `trade_calendar`
- `cb_daily`
- `macro_daily`
- `requested_codes`
- `data_quality_hints`

其中：

- `cb_daily`
  - 用于因子打分。
  - 必须包含 `FactorEngine` 所需历史列。
- `macro_daily`
  - 用于环境打分。
  - 必须覆盖环境预热所需历史。
- `requested_codes`
  - 仅控制最终展示或裁切。
  - 不能影响当日横截面标准化口径。

也就是说，和现在因子导出一样，策略计算仍然应基于全市场可交易池的当日横截面，而不是请求子集自比。

## 8. 第一阶段的组合构建口径

建议第一阶段先用最稳的规则，不要一开始就做复杂优化器。

### 8.1 组合输入

组合输入来自当日单债总分表，至少包括：

- `cb_code`
- `total_score`
- `eligible`
- `exclude_reason`

### 8.2 默认组合规则

建议第一阶段默认规则：

1. `eligible == True` 的债进入候选池。
2. 取 `total_score` 前 `top_n`。
3. 只对入选债分配权重。
4. 单债权重不超过 `single_name_max_weight`。
5. 总权重归一化到 `1 - cash_buffer`。

### 8.3 异常场景回退

需要预先定义回退规则：

- 候选池为空：返回空组合。
- 候选池少于 `min_names`：允许不足额持有，但必须在 diagnostics 里提示。
- 所有入选债分数相同：回退到等权。
- 权重截断后无法归一：回退到 capped equal weight。

## 10. 和当前导出层的关系

第一阶段建议明确区分：

- 导出层
  - 面向用户展示。
- strategy 层
  - 面向策略主链路。

因此不建议：

- 让 `strategy` 直接依赖导出层 dataclass。
- 让 `strategy` 直接复用导出层私有 helper。

但建议抽取共享逻辑，例如：

- 历史窗口起点解析
- 环境预热所需历史解析
- 因子历史窗口解析

这些能力后续更适合放到共享 helper，而不是继续留在导出层内部。

## 11. 第一阶段实现顺序

建议实现顺序如下：

### Step 1

先新增 `strategy` 目录和基础 dataclass：

- `snapshot.py`
- `result.py`

### Step 2

实现 `portfolio.py`，把“总分 -> 组合”的逻辑独立出来。

### Step 3

实现纯核心 `StrategyEngine`，先让它消费已经准备好的 snapshot。

### Step 4

实现 `StrategyService`，负责：

- 解析历史窗口
- 构造 snapshot
- 调用 `StrategyEngine`

### Step 5

补基础测试，再考虑补一个轻量的策略预览入口。

## 12. 第一阶段测试要求

建议新增：

- [tests/test_strategy_snapshot.py](C:/Users/ai/Desktop/可转债多因子/tests/test_strategy_snapshot.py)
- [tests/test_strategy_engine.py](C:/Users/ai/Desktop/可转债多因子/tests/test_strategy_engine.py)
- [tests/test_strategy_portfolio.py](C:/Users/ai/Desktop/可转债多因子/tests/test_strategy_portfolio.py)

核心测试点：

1. 同一个 snapshot 多次运行，输出完全一致。
2. 组合权重和为 `1.0` 或 `1 - cash_buffer`。
3. 单债权重不超过上限。
4. 候选为空时安全回退为空组合。
5. 请求子集不会改变全市场横截面标准化口径。
6. 修改 `trade_date` 之后的数据，不会影响当日决策。
7. `StrategyService` 走现有缓存层时，不直接依赖导出层。

## 13. 需要提前注意的工程风险

### 13.1 历史窗口逻辑不要继续散在 export 层

当前环境/因子导出里已经有一部分窗口推导 helper。`strategy` 开始后，如果继续复制一份，会形成第三套口径。

这部分建议在实现时顺手抽成共享 helper。

### 13.2 strategy 不要直接操作缓存细节

缓存命中、聚合 profile、panel cache 都应继续收在 `DataLoader / DataCacheService` 一侧。

`strategy` 最多声明“我需要什么窗口、什么列、什么 profile”，不应自己操作缓存文件。

### 13.3 第一阶段不要把 risk / execution 混进去

如果第一阶段就把风控、调仓、交易成本一起做，定位问题会很慢，也不利于先验证“纯信号主链是否成立”。

## 14. 本轮建议结论

建议按下面的方式进入 `strategy` 开发：

1. 先做 `StrategyService + StrategyEngine + StrategySnapshot + StrategyDecision` 这一条最小骨架。
2. 先做“纯信号版 portfolio”，只负责当日目标组合。
3. 暂不把执行层、风控层、回测层揉进来。
4. 实现时顺手把历史窗口解析从导出层抽成共享 helper。

这样做的好处是：

- 能尽快把主链打通。
- 能最大化复用现有 env/factor/cache 能力。
- 后续接 `backtest` 时不需要推倒重来。
