# 2026-04-21 共享历史窗口 helper 方案

## 1. 背景

截至 `2026-04-21`，环境/因子导出层和 `strategy` 层都已经具备可运行能力，但“历史窗口解析”已经出现了两套实现：

- 导出层：
  - [scoring_exports.py](C:/Users/ai/Desktop/可转债多因子/scoring_exports.py)
- `strategy` 层：
  - [strategy/service.py](C:/Users/ai/Desktop/可转债多因子/strategy/service.py)

当前重复点主要包括：

- 因子预热历史长度的推导
- 环境/因子历史请求起点的解析
- 历史实际可用起点的汇总
- “预热窗口未完全覆盖”的提示文本

这会带来两个问题：

1. 后续只要改一次口径，就要在两个地方同步修改。
2. 现在已经出现细小偏差风险，例如因子历史长度在导出层和 `strategy` 层并不是完全同一套实现。

因此，本轮需要先把“历史窗口解析”从导出层抽成共享 helper，再让 `strategy` 和导出层都走同一套口径。

## 2. 目标

本轮目标不是做新的业务规则，而是把已有规则收口成共享能力：

- 环境导出历史窗口
- 因子导出历史窗口
- `strategy` snapshot 历史窗口
- 历史起点不足时的统一提示

## 3. 本轮范围

本轮只做：

- 新增一个共享 history-window helper 模块
- 将 [scoring_exports.py](C:/Users/ai/Desktop/可转债多因子/scoring_exports.py) 中与历史窗口解析直接相关的 helper 挪过去
- 将 [strategy/service.py](C:/Users/ai/Desktop/可转债多因子/strategy/service.py) 改为复用共享 helper
- 为共享 helper 补直接测试

## 4. 本轮不做什么

本轮不纳入：

- 环境 warm-up 首日裁切逻辑重写
- 环境/因子 coverage note 重构
- summary 字段变更
- `strategy` 回测或批量预览
- 缓存层重构

## 5. 建议模块位置

建议新增：

- [history_windows.py](C:/Users/ai/Desktop/可转债多因子/history_windows.py)

放在项目根目录，而不是 `strategy/` 或 `env/` 下，原因是它服务的是“导出层 + strategy 层”的共同能力，不属于单一业务模块。

## 6. 建议收口的能力

### 6.1 因子历史长度推导

收口：

- `recommended_factor_history_buffer_calendar_days(config)`

语义保持和导出层一致：

- 由因子窗口参数动态推导
- 同时受 `exports.factor_history_buffer_calendar_days` 上限约束

### 6.2 环境导出历史起点解析

收口两层语义：

1. 环境 warm-up 需要的最早历史起点
   - 用于确保 `credit_spread` 等 reference 覆盖足够的预热区间
2. 环境导出实际请求历史起点
   - 由配置 buffer 和本地缓存起点共同决定

建议拆成两个函数，而不是混成一个：

- `resolve_environment_warmup_history_start(...)`
- `resolve_environment_report_history_start(...)`

### 6.3 因子导出历史起点解析

收口：

- `resolve_factor_report_history_start(...)`

职责：

- 先按动态因子窗口推导历史长度
- 再在 `refresh=False` 下结合本地 factor cache 起点做收缩

### 6.4 strategy snapshot 历史起点解析

收口：

- `resolve_strategy_snapshot_history_start(...)`

职责：

- 统一按：
  - `strategy.history_buffer_calendar_days`
  - `exports.env_history_buffer_calendar_days`
  - `recommended_factor_history_buffer_calendar_days(config)`
- 三者中的最宽窗口决定 snapshot 请求起点

### 6.5 历史可用起点与提示

收口：

- `safe_min_timestamp(...)`
- `max_available_history_start(...)`
- `build_history_notes(...)`

这样导出层和 `strategy` 层都能共用“请求起点 vs 实际起点”的同一套提示语义。

## 7. 保留在导出层的逻辑

下面这些仍然留在 [scoring_exports.py](C:/Users/ai/Desktop/可转债多因子/scoring_exports.py)，本轮不抽：

- 环境 `warmup_first_ready_date` 的推导
- `trend_first_ready_date` 的推导
- 请求窗口 coverage note
- summary item/comment 生成
- 环境 warm-up 说明文案

原因是这些逻辑仍然明显属于“导出语义”，不是通用历史窗口解析。

## 8. 测试要求

建议新增：

- [tests/test_history_windows.py](C:/Users/ai/Desktop/可转债多因子/tests/test_history_windows.py)

至少覆盖：

1. 因子历史长度推导遵守动态窗口和配置上限。
2. 环境导出历史起点可被本地缓存起点收缩。
3. 因子导出历史起点可被本地缓存起点收缩。
4. 环境 warm-up 起点能按 prior trade days 反推。
5. `strategy` snapshot 起点按三类 buffer 的最宽窗口决定。
6. 历史提示文本在 `used_start > requested_start` 时才出现。

## 9. 结论

本轮最稳的做法不是继续让 `strategy` 在自己模块里复制导出层 helper，而是先抽一层共享的 history-window helper。

这样后面无论我们继续推进：

- `strategy` 批量预览
- 调仓流程
- 回测主链

还是继续调整环境/因子的预热口径，都能只维护一套历史窗口解析规则。
