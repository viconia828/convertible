## 总分链路

### 1. 原始因子是怎么来的

`FactorEngine` 会先从历史日线中生成几个中间量，见 [factor_engine.py](C:/Users/ai/Desktop/可转债多因子/factor/factor_engine.py:324)：

- `momentum_60 = close / close.shift(60) - 1`
- `volatility_60 = 60 日窗口的日收益率波动率年化`
- `amount_mean_20 = 20 日平均成交额`
- `listing_obs = 当前累计交易观察天数`

然后在 snapshot 上生成 5 个原始因子，见 [factor_engine.py](C:/Users/ai/Desktop/可转债多因子/factor/factor_engine.py:434)：

- `double_low = close + premium_rate`
- `value_raw = -double_low`
- `carry_raw = ytm`
- `structure_raw = exp(-decay * ((premium_rate - center) / width)^2)`
- `trend_raw = momentum_60`
- `stability_raw = volatility_60`

### 2. eligible 现在是怎么判的

是否可建仓仍然由这组布尔条件共同决定，见 [factor_engine.py](C:/Users/ai/Desktop/可转债多因子/factor/factor_engine.py:497)：

- 不是 `recently_listed`
- `remain_size` 达标
- `amount_mean_20` 达标
- 不是 `call_announced`
- 不是 `put_triggered`
- 当日可交易
- `has_required_fields = True`

本次用到的关键阈值来自 [策略参数.txt](C:/Users/ai/Desktop/可转债多因子/策略参数.txt:125)：

- `min_listing_days = 30`
- `min_remain_size = 50000000.0`
- `min_avg_amount_20 = 200.0`
- `momentum_window = 60`
- `volatility_window = 60`

但 `has_required_fields` 的口径已经不是旧版那种“必须 `momentum_60` 非空”，而是改成了两段式，见 [factor_engine.py](C:/Users/ai/Desktop/可转债多因子/factor/factor_engine.py:444)：

- 基础必需字段仍然要齐：`premium_rate / ytm / volatility_60`
- 如果 `momentum_60` 已经算出来，就按正常趋势因子处理
- 如果 `momentum_60` 还没算出来，但样本已经过了 `30` 日观察期、仍处在 `60` 日动量预热段内，那么会打上 `uses_trend_neutral_score = True`
- 这类样本会被视为 `has_required_fields = True`，允许进入 `eligible` 池

换句话说，当前逻辑把样本分成三类：

- `listing_obs < 30`：还是新券，照旧排除
- `30 < listing_obs <= 60` 且只缺 `momentum_60`：不排除，趋势给中性分
- `listing_obs > 60` 但仍缺关键字段：继续按 `missing_required_fields` 排除

### 3. total_score 是怎么来的

只有 `eligible=True` 的样本才会进入横截面打分，见 [factor_engine.py](C:/Users/ai/Desktop/可转债多因子/factor/factor_engine.py:461)：

- `value_score`、`carry_score`：winsorize 后做 `z-score`
- `structure_score`：做百分位
- `trend_score`：
  - 正常样本：对 `trend_raw` 做百分位
  - `uses_trend_neutral_score=True` 的样本：直接给 `0.5`
- `stability_score`：`1 - 波动率百分位`

趋势中性分的赋值逻辑见 [factor_engine.py](C:/Users/ai/Desktop/可转债多因子/factor/factor_engine.py:489)。

随后 `StrategyEngine` 用当天环境映射出的因子权重做加权求和，见 [engine.py](C:/Users/ai/Desktop/可转债多因子/strategy/engine.py:65) 和 [factor_engine.py](C:/Users/ai/Desktop/可转债多因子/factor/factor_engine.py:258)。

这次运行当天实际权重是：

- `value = 0.262950`
- `carry = 0.217061`
- `structure = 0.200332`
- `trend = 0.174491`
- `stability = 0.145166`

注意：非 `eligible` 样本的 `total_score` 会被直接置空，见 [factor_engine.py](C:/Users/ai/Desktop/可转债多因子/factor/factor_engine.py:271)。

## 样本二：`31~60` 日预热期、使用趋势中性分的 `110100.SH`

### 关键现象

- `listing_obs = 41`
- `premium_rate = 60.13`
- `ytm = -0.046444`
- `momentum_60 = NaN`
- `volatility_60 = 0.257682`
- `is_tradable_now = True`
- `is_size_ok = True`
- `is_amount_ok = True`

### 为什么现在不再被排除

它在旧口径里会因为 `momentum_60 = NaN` 被打成 `missing_required_fields`。

但在当前口径下：

- 它已经过了 `min_listing_days = 30`，所以不是 `recently_listed`
- `premium_rate / ytm / volatility_60` 都齐，所以基础字段是够的
- `momentum_60` 虽然还没出来，但它当前 `listing_obs = 41`，仍在 `30~60` 的动量预热区间
- 因此它会被打上 `uses_trend_neutral_score = True`
- 进而 `has_required_fields = True`
- 最后 `eligible = True`

它这次的趋势相关结果是：

- `trend_raw = NaN`
- `trend_score = 0.5`

注意这里是一个很关键的新口径：

- `trend_raw` 允许继续是空值
- 但 `trend_score` 会被补成中性分 `0.5`
- 这不是数据脏了，而是策略有意为之

它这次的综合结果是：

- `total_score = 0.399201`
- 候选排序约第 `100` 位
- 已进入 `eligible` 池，但并没有进入前 `15`

所以当前 `110100.SH` 不再是“必需字段不足”的代表样本，而是“趋势因子预热期中性分放行”的代表样本。

- 更像“候选池扩容 + 早期纳入观察”，而不是“直接增强短期趋势权重”

## 当前未出现的两类特殊情况

### 1. `put_triggered`

本次 `factor_diagnostics` 里没有任何 `put_triggered=True` 的样本。

更重要的是，当前实现里这一列还没有真正接入数据判断逻辑；它在 enrich 阶段被直接初始化为 `False`，见 [factor_engine.py](C:/Users/ai/Desktop/可转债多因子/factor/factor_engine.py:420)。

这意味着：

- 但当前主链路还不会真实产出 `put_triggered` 排除案例

### 2. `missing_daily_history`

本次导出没有观察名单，所以也没有 `missing_daily_history`。

这类情况只会在“用户显式指定了观察名单代码，但主数据里没有找到该券的日线记录”时，由占位逻辑补出来，见 [factor_engine.py](C:/Users/ai/Desktop/可转债多因子/factor/factor_engine.py:636)。

这种占位样本会被写成：

- `eligible = False`
- `exclude_reason = missing_daily_history`
- 其余诊断列按缺失占位处理

它更像“观察名单补位提示”，不属于正常全市场打分链路里的业务排除。

## 对照 xlsx 的阅读建议

如果你边看表边核对，建议顺序是：

1. 先看 `factor_weights` 页，确认当天 5 个因子权重
2. 再看 `factor_diagnostics` 页，先判断某只券 `eligible` 为什么是 `True/False`
3. 如果 `eligible=True` 且 `uses_trend_neutral_score=True`，要把它理解成“趋势中性分放行”，不要误判成数据缺失
4. 如果 `eligible=True`，再看 5 个因子分数如何加总成 `total_score`
5. 如果 `eligible=False`，优先看 `exclude_reason` 和那几个布尔标记，不要先盯 `total_score`

因为当前实现里，非 `eligible` 样本本来就不会再继续算分并参与排序；而对 `31~60` 日新券来说，是否被放行的关键已经不再是 `momentum_60` 本身，而是 `uses_trend_neutral_score` 这个口径标记。
