# 2026-04-24 缓存 schema / 口径版本治理方案

## 1. 背景

当前缓存层已经具备：

- 统一缓存门面 `DataCacheService`
- 运行期 `request panel / aggregate / aggregate metadata` 复用
- `coverage.json` 和 aggregate `.meta.json` 两类元数据

但这些元数据目前还缺一层关键能力：

- 无法识别“当前代码口径”和“当年写缓存时的口径”是否一致

这会带来两类隐患。

### 1.1 schema 变化后，旧 metadata 仍可能被继续消费

例如：

- `DataSchema` 增减字段
- 字段 dtype 变化
- key / sort 列变化

当前 metadata 没有任何 schema 指纹，无法知道这些变更是否已经让旧 metadata 失效。

### 1.2 aggregate metadata 只校验日期覆盖，不校验列投影兼容性

当前 `cb_daily_cross_section` 的 aggregate metadata 虽然已经保存了：

- `covered_trade_days`
- `projection_columns`

但读路径里目前真正用到的只有：

- `covered_trade_days`

也就是说：

- 如果某个月的聚合缓存是按旧列集合建出来的
- 现在请求又需要更多列

系统并不会明确识别“这份 aggregate metadata 和当前请求不兼容”。

## 2. 本轮目标

第一版版本治理只解决两个最有价值的问题：

1. 为现有 `coverage.json` 和 aggregate `.meta.json` 增加统一的治理块。
2. 在读取这些 metadata 时，显式判断：
   - schema 指纹是否仍兼容
   - aggregate 的列投影是否仍兼容当前请求

如果不兼容，则统一按“metadata 无效”处理，让上层自然走重建路径。

## 3. 本轮不做

本轮不做：

- 不为所有 CSV 缓存文件新增 sidecar metadata
- 不引入跨进程的复杂迁移脚本
- 不做多版本并存管理
- 不在第一版就治理 `credit_spread` reference metadata
- 不把 request panel runtime memory 做成带版本号的持久化缓存

## 4. 第一版治理范围

### 4.1 纳入治理

- `time_series_coverage` metadata
- `time_series_aggregate_metadata`

### 4.2 暂不纳入治理

- static CSV
- 普通 per-code time-series CSV
- reference CSV
- request panel runtime memory

原因很简单：

- 现有系统里真正已经有 metadata 的，就是这两类
- 它们正好也是“高层缓存是否继续可用”的判定入口

## 5. 设计思路

### 5.1 为标准化数据集生成 schema 指纹

基于 `DataSchema` 中的标准化定义，生成稳定 schema 指纹，至少覆盖：

- dataset name
- key columns
- sort columns
- date columns
- fields:
  - local_name
  - source_name
  - dtype
  - mutability

只要这些定义变了，schema 指纹就会跟着变。

### 5.2 为 metadata 增加统一治理块

在 metadata 中新增统一字段，例如：

```json
{
  "governance": {
    "metadata_format_version": 1,
    "standardized_name": "cb_daily",
    "schema_signature": "..."
  }
}
```

aggregate metadata 在此基础上额外记录：

- `profile`
- `projection_signature`

## 6. aggregate metadata 的兼容规则

aggregate metadata 兼容，至少要同时满足：

1. `governance` 存在
2. `metadata_format_version` 匹配
3. `standardized_name` 匹配
4. `schema_signature` 匹配当前 `DataSchema`
5. 当前请求列集合是 `projection_columns` 的子集

最后一条很关键：

- 如果旧聚合缓存保存的是列集合 A
- 当前请求列集合是 A 的子集
  - 可以继续复用
- 当前请求列集合包含了 A 中没有的新列
  - 必须视为 metadata 不兼容，并回退重建

## 7. coverage metadata 的兼容规则

coverage metadata 兼容，至少要满足：

1. `governance` 存在
2. `metadata_format_version` 匹配
3. `standardized_name` 匹配
4. `schema_signature` 匹配

对于第一版来说：

- 一旦缺 governance，就按 legacy / stale metadata 处理
- 直接返回 `None`

这样上层会自然走“重新补 coverage”路径。

## 8. legacy metadata 的处理策略

第一版对 legacy metadata 的策略是：

- 不做原地迁移
- 不尝试猜兼容
- 直接按“不兼容 / 无效 metadata”处理

这样会带来的行为是：

- 升级后的第一轮运行里
  - aggregate metadata 可能被忽略一次
  - sparse coverage metadata 可能被忽略一次
- 系统随后会按当前代码口径重新生成新 metadata

这个成本是一次性的，而且明显低于“继续默默吃旧 metadata”带来的风险。

## 9. 预期收益

这轮版本治理完成后，至少可以保证：

1. `DataSchema` 变更后，高层 metadata 不会再无感继续复用。
2. aggregate metadata 不会再只看日期覆盖，而忽略列投影不兼容。
3. 后续如果继续做 schema / 口径治理，这一层已经有统一挂点。

## 10. 实现位置

建议新增：

```text
data/cache/governance.py
```

集中承接：

- metadata format version 常量
- schema signature 生成
- projection signature 生成
- coverage metadata payload 构造 / 校验
- aggregate metadata payload 构造 / 校验

然后在 `DataCacheService` 中接入：

- `save_time_series_coverage`
- `load_time_series_coverage`
- `save_time_series_aggregate_metadata`
- `load_time_series_aggregate_metadata`

再由 `DataLoader` 的 aggregate 读取路径显式消费：

- aggregate metadata compatibility check

## 11. 验证标准

完成后至少应满足：

1. 新写入的 coverage / aggregate metadata 都带有治理块。
2. schema 指纹不匹配时，metadata 会被判定为无效。
3. aggregate metadata 的列投影不兼容时，会被判定为无效。
4. 现有 `tests/test_cache_service.py`、`tests/test_data_loader.py` 回归通过。
5. 旧 metadata 缺 governance 时，不会误判为兼容。

## 12. 完成后的下一步

如果这轮第一版稳定，下一步可以继续评估：

1. 是否把 reference metadata 也接入同一套治理块。
2. 是否为更底层的 CSV cache 引入轻量 sidecar。
3. 是否把“schema / 口径版本”进一步显式配置化，而不是只依赖 schema 指纹。
