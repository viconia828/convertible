# 2026-04-21 统一缓存层设计方案

## 1. 背景

当前项目已经有本地缓存能力，但缓存文件与缓存策略并没有完全收口：

- `data/cache_store.py` 主要负责文件路径和 CSV 读写，属于底层存储层
- `data/data_loader.py` 承担了大量缓存协调职责，包括覆盖判断、批量读取、缺口补取、coverage metadata、派生字段回写等
- `data/trading_calendar.py` 维护了自己的 `cache -> remote -> fallback` 逻辑
- `scoring_exports.py` 仍然直接探测本地缓存可用起点

这种结构会让性能优化、命中观察、策略调整都越来越分散，不利于后续继续做缓存升级。

## 2. 目标

在数据层和环境层 / 因子层之间增加统一缓存层，收口以下职责：

- 缓存文件存取
- 时间覆盖判断和稀疏事件 coverage metadata
- 批量缓存读取
- 派生字段回写
- 本地可用历史起点探测
- 缓存读写统计与后续观测扩展入口

目标边界是：

1. 数据源层只负责远端取数和 schema 标准化
2. 缓存层统一负责命中、补缺、合并、回写和观测
3. `DataLoader` 只保留 dataset 语义和对外稳定接口
4. 环境层 / 因子层 / 导出层不再直接碰缓存细节

## 3. 本次需要统一收口的已有策略

第一阶段优先收口这些已有逻辑：

- `cb_basic` fixed / mutable 缓存
- 通用 time-series 覆盖判断
- `cb_call` 的 coverage metadata
- `cb_rate` / `cb_daily_cross_section` 的批量缓存读取
- `ytm` 等派生字段回写
- 环境层和因子层依赖的本地历史起点探测
- `TradingCalendar` 的缓存 / fallback 决策

## 4. 目标模块结构

第一阶段先落一个可工作的统一入口：

```text
data/cache/
  __init__.py
  service.py
```

其中：

- `service.py` 提供 `DataCacheService`
- `DataCacheService` 统一承接缓存协调逻辑
- `CacheStore` 继续保留为底层文件读写组件

后续如果职责继续增多，再把 `service.py` 继续拆为：

```text
data/cache/
  __init__.py
  service.py
  observer.py
  coverage.py
  writeback.py
  policy.py
  models.py
```

## 5. 第一阶段统一接口

当前统一入口由 `DataCacheService` 提供，已经覆盖：

- `load_calendar` / `save_calendar`
- `load_static_frame` / `save_static_frame`
- `load_time_series` / `save_time_series`
- `load_reference_frame`
- `load_grouped_time_series`
- `covers_time_series`
- `covers_expected_dates`
- `covers_sparse_range`
- `load_time_series_coverage` / `save_time_series_coverage`
- `writeback_derived_fields`
- `inspect_local_env_history_start`
- `inspect_local_factor_history_start`
- `stats_snapshot`

## 6. 已落地的第一阶段实现

本设计文档对应的第一阶段实现已经完成，具体包括：

- 新增 `data/cache/__init__.py`
- 新增 `data/cache/service.py`
- `DataLoader` 改为优先通过 `cache_service` 处理：
  - static cache
  - grouped cache load
  - coverage metadata
  - derived writeback
  - local ready range inspect
- `TradingCalendar` 改为通过 `cache_service` 处理缓存命中和 fallback
- `scoring_exports.py` 不再直接读取 `cache_store`，改为调用缓存层探测本地历史起点

## 7. 当前边界

### 数据层保留

- 远端 API 参数拼装
- schema 标准化
- dataset 语义定义

### 缓存层承接

- 命中与覆盖判断
- 批量缓存读取
- coverage metadata
- 派生字段回写
- 本地可用范围探测

### 环境层 / 因子层 / 导出层

- 只消费数据结果
- 不再直接处理缓存文件路径和命中细节

## 8. 后续优化分支

统一缓存层收口后，后续优化建议按下面顺序展开：

### 8.1 观测能力

- 增加 hit / miss / partial hit 统计
- 记录文件扫描数、远端补数次数、回写次数
- 输出阶段耗时，给 benchmark 和调参提供统一口径

### 8.2 缓存格式升级

- 评估将 `cb_daily_cross_section` 从按日 CSV 升级为聚合缓存
- 评估将 `cb_rate` 从按代码小文件升级为更高效的聚合格式
- 视情况评估切换到 `parquet`

### 8.3 派生指标缓存

- 将 `ytm` 等派生值升级为独立 derived cache
- 区分原始缓存和派生缓存，降低对原表的频繁回写

### 8.4 运行期缓存

- 增加单次运行内的内存级缓存
- 减少同一轮运行中的重复 `read_csv`

### 8.5 版本和失效治理

- 按 schema 版本管理缓存失效
- 为 derived cache 增加口径版本

## 9. 第一阶段验收标准

完成后应满足：

1. `DataLoader` 不再直接承载主要缓存策略实现
2. `TradingCalendar` 和导出层不再直接探测底层缓存文件
3. 批量缓存读取、coverage metadata、派生回写都有统一入口
4. 现有回归测试继续通过
5. 后续继续做观测、格式升级、derived cache 时，不需要再改业务层接口
