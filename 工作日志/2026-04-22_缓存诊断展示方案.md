# 2026-04-22 缓存诊断展示方案

## 背景

前一轮已经把统一缓存层的观测指标补齐：

- `hit / miss / partial hit / refresh bypass`
- 文件扫描数
- 远端补数次数
- 回写次数
- 分阶段耗时

同时也保留了原有更细的内存层统计：

- `panel_memory_*`
- `aggregate_memory_*`
- `aggregate_metadata_memory_*`

但这些信息目前还主要停留在底层计数和 benchmark 原始 JSON 里，终端入口还缺一层更统一、更便于人工判断的诊断展示。

## 目标

这一轮只解决“展示和消费”问题，不再改底层缓存判定口径。

希望做到：

1. `strategy` 预览能直接看到本次运行的缓存诊断，而不是进程累计值。
2. `strategy` 预览能显式展示 `runtime snapshot / request panel / aggregate` 三层收益。
3. `benchmark` 输出能更直接展示每次迭代的缓存收益变化，便于验证复跑收益。
4. 展示口径集中到共享 helper，避免 `preview` 和 `benchmark` 各自拼一套字符串。

## 范围

本轮纳入：

- 新增共享 `cache_diagnostics` helper
- `StrategyService` 计算“单次运行前后”的 cache stats delta
- `StrategySnapshot` / `StrategyDiagnostics` 透传结构化 cache diagnostics
- `preview_strategy.py` 展示缓存诊断区块
- `benchmark_factor_export.py` 输出累计 + 每轮迭代 cache diagnostics
- 对应测试、README、工作日志和下一步清单同步

本轮不做：

- 不新增新的底层统计口径
- 不修改 `DataLoader` 补数策略
- 不新增 UI 面板
- 不把 benchmark JSON 改成纯文本模式，仍保留结构化输出

## 设计

### 1. 共享 helper 负责三件事

新增 `cache_diagnostics.py`，集中提供：

- `diff_cache_stats(...)`
  - 计算一次运行前后的 cache stats 增量
- `build_cache_diagnostics(...)`
  - 把原始 stats 收口成更易消费的结构化诊断
  - 包括：
    - 统一观测摘要
    - `request_panel / aggregate / aggregate_metadata` 三层命中摘要
    - 主要阶段耗时 top N
    - 可选 `runtime_snapshot_reused`
- `render_cache_diagnostic_lines(...)`
  - 为终端入口生成紧凑文本行

### 2. strategy 预览展示“本次运行”的 cache 诊断

问题在于：

- `DataCacheService.stats_snapshot()` 是同一进程累计值
- 直接塞进 `StrategyDecision` 会导致第二次 runtime reuse 预览看到第一次构建留下的累计统计

所以 `StrategyService.build_snapshot(...)` 要在本次构建前后各抓一次 stats：

- fresh build:
  - `delta = after - before`
  - 用 `delta` 构造 `cache_diagnostics`
- runtime snapshot reuse:
  - 不再触发 loader
  - 本次 `delta` 应该是零
  - 但仍要显式标记 `runtime_snapshot_reused=True`

这样预览展示的是“当前这一轮 build 的 cache 行为”，而不是整个进程从启动到现在的累计值。

### 3. preview 里展示两层信息

摘要模式：

- 一行显示 `runtime snapshot / request panel / aggregate`
- 一行显示 `hit / miss / partial hit / remote fill / file scan / writeback`

详细模式：

- 保留上述摘要
- 额外显示 `aggregate metadata`
- 额外显示 top 阶段耗时

### 4. benchmark 输出展示累计值和迭代值

`benchmark_factor_export.py` 继续保留：

- `cache_stats`
- `cache_observability`

同时新增：

- `cache_diagnostics`
  - 全部 benchmark 运行完成后的累计诊断
- `iteration_cache_diagnostics`
  - 每次 repeat 的增量诊断

这样就能更直观看到：

- 第一轮是否以 miss / remote fill 为主
- 第二轮及之后是否已经转为 panel / aggregate hit

## 风险与取舍

### 1. preview 不直接展示原始 `cache_stats`

原因：

- 原始计数过碎
- 对终端阅读不友好
- 也容易把“当前运行”与“累计值”混在一起

所以 preview 只展示结构化摘要，不展示完整原始 dict。

### 2. benchmark 继续保留 JSON 输出

原因：

- 现有 benchmark 已经是结构化 JSON
- 直接改成纯文本会增加兼容风险

因此本轮是在 JSON 里新增更适合人读的 `cache_diagnostics` 字段，而不是替换原输出格式。

## 验证

至少覆盖：

1. `diff_cache_stats(...)` 与结构化 cache diagnostics 汇总
2. `preview_strategy` 摘要 / 详细模式都能显示缓存诊断
3. `StrategyService` 第二次 runtime reuse 时，cache diagnostics 为“本次零增量 + runtime reuse 命中”
4. `benchmark` 的迭代诊断能反映每轮 stats 增量
