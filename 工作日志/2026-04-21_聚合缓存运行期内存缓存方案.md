# 2026-04-21 聚合缓存运行期内存缓存方案

## 1. 背景

在 [2026-04-21_cb_daily_cross_section聚合缓存方案.md](C:/Users/ai/Desktop/可转债多因子/工作日志/2026-04-21_cb_daily_cross_section聚合缓存方案.md) 落地后，`cb_daily_cross_section` 的最热读取路径已经从“按日小文件”切到了“按月聚合分片”。

当前收益主要体现在：

- 首次运行负责建立 `factor_history_v1` 月度聚合缓存
- 同窗口或重叠窗口再次运行时，可以少读大量按日 CSV

但在同一 Python 进程内，如果连续多次运行因子打分，当前实现仍会：

- 反复读取同一批月度聚合 CSV
- 反复解析同一批 `.meta.json`
- 反复做同样的标准化与筛选

这说明聚合缓存之上还缺一层“运行期内存缓存”。

## 2. 本轮目标

在统一缓存层内部，为聚合缓存增加一层轻量运行期内存缓存，目标是：

- 让同一进程内的重复构建优先命中已解析的聚合分片
- 继续把缓存策略收口在 [service.py](C:/Users/ai/Desktop/可转债多因子/data/cache/service.py)
- 不改变上层业务接口和结果口径

## 3. 设计范围

本轮只覆盖两类对象：

- 聚合缓存 frame
- 聚合缓存 metadata

本轮不做：

- 普通按日 / 按代码 time-series 的通用运行期内存缓存
- 跨进程共享内存缓存
- 更复杂的对象池或 mmap 方案

## 4. 关键设计

### 4.1 缓存位置

运行期内存缓存只放在 `DataCacheService` 内部，不暴露给业务层。

这样：

- `DataLoader` 仍然只表达“我要哪段数据”
- `scoring_exports` 仍然只表达“我要哪个 profile”
- 缓存命中、失效、观测统一留在缓存层

### 4.2 缓存对象

#### 聚合 frame

frame 缓存 key 包含：

- `dataset_name`
- `profile`
- `partition_key`
- `standardized_name`
- `requested_columns`

说明：

- `requested_columns` 需要做归一化，避免同一组列因顺序或重复而出现多份缓存
- frame 在进入运行期内存缓存前，先完成标准化

#### 聚合 metadata

metadata 缓存 key 包含：

- `dataset_name`
- `profile`
- `partition_key`

### 4.3 缓存策略

本轮使用轻量 LRU 策略：

- frame cache 只保留少量最近使用分片
- metadata cache 保留稍多一些最近使用项

原因：

- 同一轮因子打分通常只涉及有限个月份
- 这样可以控制进程内常驻内存，不把“轻量缓存”做成无上限堆积

### 4.4 读路径

对于 `load_time_series_aggregate(...)`：

1. 先查运行期 frame cache
2. 命中则直接返回共享只读对象，并记录 memory hit
3. 未命中再读磁盘 CSV
4. 读盘后完成标准化，写入运行期 frame cache，再返回共享只读对象

对于 `load_time_series_aggregate_metadata(...)`：

1. 先查运行期 metadata cache
2. 命中则直接返回共享只读对象，并记录 memory hit
3. 未命中再读 `.meta.json`
4. 读盘后写入运行期 metadata cache，再返回共享只读对象

说明：

- 这里的“共享只读对象”是缓存层内部优化约定
- 当前上层调用链只按只读方式消费这些对象
- 这样可以避免命中内存缓存后仍因整表深拷贝抵消收益

### 4.5 写路径与失效

对于 `save_time_series_aggregate(...)`：

- 写盘后，清理对应分片的旧 frame memory cache
- 同时把当前写入结果直接写入运行期 frame cache，作为新值

对于 `save_time_series_aggregate_metadata(...)`：

- 写盘后，清理对应 metadata memory cache
- 同时把当前 payload 直接写入运行期 metadata cache，作为新值

这样可以保证：

- 同一进程内后续读取不会看到旧值
- 刚刚物化出的月度聚合缓存可以立即被下一次构建复用

## 5. 观测指标

在 `DataCacheService.stats_snapshot()` 中新增至少以下统计：

- `aggregate_memory_hit_calls`
- `aggregate_memory_miss_calls`
- `aggregate_metadata_memory_hit_calls`
- `aggregate_metadata_memory_miss_calls`

并继续保留 dataset / profile 级细分统计，便于后续判断：

- 哪个 profile 最常命中运行期内存缓存
- 哪个 profile 仍主要消耗在磁盘层

## 6. 对 benchmark 的配套调整

为了观察“同一进程内重复构建”的收益，benchmark 工具增加重复运行能力：

- 新增 `--repeat`
- 默认仍为 `1`
- 当 `repeat > 1` 时，复用同一个 `DataLoader`

这样可以直接观察：

- 第一次运行：可能承担磁盘聚合缓存命中或聚合缓存物化
- 第二次及之后运行：是否稳定命中运行期内存缓存

## 7. 验证标准

完成后应满足：

1. 同一进程内重复构建时，聚合 frame / metadata 出现明确的 memory hit
2. `save_time_series_aggregate(...)` 后，同进程再次读取可见新值
3. 结果口径不变，现有回归测试继续通过
4. benchmark 能输出运行期内存缓存命中统计
5. 同进程重复构建的热路径耗时继续下降

## 8. 预期收益与后续空间

这轮优化的主要收益不一定体现在“冷启动首次运行”，而是体现在：

- 同一进程内连续多次导出
- 同一线程内反复试跑参数
- 后续 `strategy` 主链路里复用同一 `DataLoader` 的场景

如果这轮验证效果稳定，下一步可以继续评估：

- 是否把运行期内存缓存扩展到更多稳定 aggregate profile
- 是否为普通 time-series 读取增加更受控的短生命周期内存层
- 是否把内存缓存 stats 接到更统一的缓存观测报告里
